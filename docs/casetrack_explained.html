1 A tracker, not a scheduler

The single most important mental model.

casetrack does not run analyses. Your pipelines do — Nextflow, Snakemake, or raw SLURM jobs execute the actual basecalling, alignment, variant calling, and methylation work. casetrack stores the facts afterward: which analyses completed, which samples are pending, which are blocked by QC.

Pipelines
Run the work
Nextflow / Snakemake / SLURM produce BAMs, VCFs, bedMethyl…
→
casetrack
Records the facts
One SQLite DB: who's done, who's pending, who's censored.
→
You / agents
Query & decide
Generate the next batch's samplesheet from the DB.

The source of truth is a single casetrack.db (SQLite, WAL-mode). The TSV manifest you may be used to becomes an on-demand export, not the master copy.

2 The 3-level hierarchy

Everything hangs off a strict, foreign-key-enforced ontology.

patients
clinical / demographic metadata — sex, cohort, timepoint, age, consent…
specimens
a biological sample — tumor / normal / cell line. Most analysis tracking lives here.
assays
an individual sequencing run — per-run basecalling & QC live here.

Registration order is fixed: patient → specimen → assay. A specimen row needs its patient_id; an assay row needs its specimen_id. Each analysis is declared to run at exactly one level:

Analysis kindLevelExamples
Per-runassaydorado basecalling, per-run QC
Per-samplespecimensamtools sort, modkit pileup, SV calling, merge
Per-patientpatientcohort summary, family-level SV

3 add-metadata vs append

The #1 source of confusion — and silent failures. Internalize this.

Ask one question: am I registering an entity, or recording that an analysis ran?

add-metadata

Builds the cohort

Populates native schema columns on patients / specimens / assays. Use it to declare who exists.

  • No --analysis flag
  • Creates new rows with --allow-new --yes
  • Columns land as-is
  • Does not write a _done timestamp
append

Records analysis results

Fills in the result columns for a tool that already ran. The cohort must already exist.

  • Requires --analysis <name> (must match TOML)
  • Cannot create new rows
  • Every result column gets a {prefix}_
  • Writes {analysis}_done = now()
The silent gotcha — --overwrite. append defaults to fill-only: an existing non-NULL cell is never updated. A rerun with fresh stats silently no-ops at the DB level with no error. Always pass --overwrite for analysis results. Fill-only is only correct when backfilling empty cells.

4 Schema = the TOML contract

Changes flow one way: TOML → DB. Never the reverse.

casetrack.toml declares the schema; the SQLite DB mirrors it. Edit the TOML, then apply — non-destructive ALTER TABLEs run for you.

# after editing [levels.*.columns] or adding [analyses.<name>]:
casetrack schema apply --project-dir .
# → "Applied N schema change(s); schema_v X → Y"
Never edit casetrack.db directly and never run ad-hoc ALTER TABLE. The TOML is the source of truth; the DB is a cached materialization. Forgetting schema apply after a TOML edit ⇒ new columns are invisible (no such column).

5 The 3-phase analysis flow

Every analysis — local, SLURM, or Nextflow — follows the same shape.

Phase 1
Run the tool
Nextflow / SLURM / local script produces outputs.
→
Phase 2
Summarize → TSV
One row per entity at the analysis's level. First column = the level's key.
→
Phase 3
casetrack append
--overwrite. Stats land prefixed; _done stamped.

Example — after samtools sort on 6 specimens, with a summary keyed on specimen_id:

# summary.tsv columns:
#   specimen_id  sorted_bam_path  sorted_bam_size_bytes  n_reads  sort_order

casetrack append --project-dir . \
  --analysis samtools_sort \
  --results /tmp/sort_summary.tsv \
  --overwrite

# prefix "sort" comes from [analyses.samtools_sort].column_prefix in TOML →
# writes: sort_sorted_bam_path, sort_n_reads, … , samtools_sort_done = now()

If the summary TSV also carries qc_pass / qc_fail_reason / qc_warn columns, casetrack auto-emits QC events inside the same transaction (v0.4+).

6 Building a cohort from scratch

The canonical bootstrap sequence.

# 1. create the project
casetrack init --project-dir /path/cohort_X --project-name cohort_X

# 2. edit cohort_X/casetrack.toml — add columns (sample_id, timepoint, pod5_path…)
#    and declare analyses: [analyses.<tool>] { level, column_prefix, summary_tsv }

# 3. apply the schema
casetrack schema apply --project-dir .

# 4–6. register entities — parents first (FK!)
casetrack add-metadata --project-dir . --level patient  --metadata patients.tsv  --allow-new --yes
casetrack add-metadata --project-dir . --level specimen --metadata specimens.tsv --allow-new --yes
casetrack add-metadata --project-dir . --level assay    --metadata assays.tsv    --allow-new --yes
Add pod5_path to your assay columns. It's what lets you drive Nextflow samplesheets straight from the DB (§7). Missing it is a common regret.

7 The DB as a work queue

casetrack already knows what's done and what's pending — so query it to build the next batch.

1
Query pending
SELECT samples WHERE {analysis}_done IS NULL AND qc_status='pass'.
→
2
→ samplesheet
Pipe to CSV; submit to Nextflow / Snakemake.
→
3
append --overwrite
DB stays current; the pending query shrinks next run.
# all normals that still need basecalling, excluding censored samples
casetrack query --project-dir . --sql "
  SELECT a.assay_id, p.patient_id, s.specimen_id, a.pod5_path, 'hg38' AS genome
  FROM assays a
  JOIN specimens s ON a.specimen_id = s.specimen_id
  JOIN patients  p ON s.patient_id  = p.patient_id
  WHERE a.condition = 'normal'
    AND a.qc_status = 'pass'
    AND a.dorado_basecaller_done IS NULL"

Resubmit any time: Nextflow -resume skips finished tasks, append --overwrite keeps the DB honest. Prefer the _active view over raw tables — it also cascades parent censoring.

8 Nextflow integration flow

Phase 3 becomes a process: CASETRACK_REGISTER.

process
TOOL_PROCESS
runs the tool
→
process
SUMMARIZE_TOOL
emits a 1-row TSV
→
process
CASETRACK_REGISTER
append --infer-from-path --overwrite
include { CASETRACK_REGISTER } from '../modules/local/casetrack_register'

workflow FOO_TRACKED {
    take: inputs
    main:
    TOOL_PROCESS(inputs)                                // Phase 1
    SUMMARIZE_TOOL(TOOL_PROCESS.out.files)              // Phase 2
    CASETRACK_REGISTER(                                 // Phase 3
        SUMMARIZE_TOOL.out.summary.map { meta, tsv ->
            tuple(meta, 'samtools_sort', 'samtools_sort_summary.tsv', tsv)
        }
    )
}

The tool name, run_tag, patient, specimen and assay_id are recovered from the publish path — no explicit flags. Required params on every run:

ParamMeaning
--casetrack_project_dirabsolute path, outside the NF work dir
--run_tag{YYYYMMDD}_{genome}_{description}
--casetrack_levelassay | specimen | patient
--casetrack_bindefaults to casetrack
CASETRACK_REGISTER runs on the local executor with maxForks = 1. SQLite is WAL-mode, but serializing writes keeps the provenance log readable.

9 QC events & censoring

Append-only audit log. Nothing is ever deleted.

censor
Open an event
writes an event + sets qc_status.
→
_active view
Hides the entity
excludes it (and cascades parent censoring).
→
uncensor
Resolve it
writes resolved_at — never deletes the row.
# temporary hold
casetrack censor --project-dir . --level assay --id <id> \
  --kind qc_warn --reason "pod5 rsync incomplete 2026-04-21"

# permanent exclusion
casetrack censor --project-dir . --level assay --id <id> \
  --kind qc_fail --reason "basecalling accuracy <80%"

# lift the hold
casetrack uncensor --project-dir . --level assay --id <id> \
  --reason "pod5 rsync confirmed complete"

QC kinds: qc_fail, qc_warn, consent_revoked, sequencing_run_failed, library_prep_failed, basecall_accuracy_low, contamination, protocol_deviation, batch_effect_flagged, superseded, other.

Entity qc_status: pass · warn · fail · censored.

consent_revoked is patient-level only and cascades to all specimens/assays under that patient. Reversing it requires --ethics-override --yes and an IRB reference in the reason.

10 Cohort-level artifacts

For one output built from many assays — joint VCFs, panels-of-normals, cohort matrices.

The 3-level hierarchy is biological, single-parent, and static — it can't represent an output that spans dozens of assays. So casetrack adds two additive sibling tables (the same pattern as qc_events — the core hierarchy is untouched):

TableRole
cohort_artifactsone row per cohort output, keyed by (analysis, run_tag)
cohort_artifact_inputsmany-to-many lineage to each contributing assay_id
Staleness is read-time, not stored. An artifact is STALE when any contributing assay is currently censored or consent-revoked — derived live from the QC cascade, so it tracks censor/uncensor automatically. Staleness is flagged, not auto-fixed: re-running is the operator's call.

Surfaced everywhere: the cohort-artifacts command, status, the _cohort_artifacts DuckDB query view, export --include-cohort-artifacts, the HTML dashboard, and the casetrack_cohort_artifacts MCP tool. (A 4th hierarchy level was explicitly rejected — see proposal 0009 §7.)

11 On-disk layout

Persistent biological files vs. per-run summaries.

{project_dir}/
├── casetrack.toml
├── casetrack.db
├── data/processed/{genome}/{patient_id}/{assay_id}/   ← persistent biology
│     {assay_id}.{genome}.sorted.bam
│     {assay_id}.{genome}.sniffles.vcf.gz
└── results/{tool}/{run_tag}/{patient_id}/{specimen_id}/{assay_id}/  ← summary TSVs

The DB stores the data/processed/ path from each summary TSV, so downstream tools find files by query rather than by filesystem scan. Run-tag convention: {YYYYMMDD}_{genome}_{description}.

12 Command cheatsheet

Always use --project-dir (project mode). --manifest is legacy flat mode — avoid it.

# project lifecycle
casetrack init --project-dir PATH --project-name NAME
casetrack schema apply --project-dir .
casetrack schema show  --project-dir .

# entity registration
casetrack add-metadata --project-dir . --level patient  --metadata p.tsv --allow-new --yes
casetrack add-metadata --project-dir . --level specimen --metadata s.tsv --allow-new --yes
casetrack add-metadata --project-dir . --level assay    --metadata a.tsv --allow-new --yes

# analysis tracking (--overwrite, always)
casetrack append --project-dir . --analysis NAME --results summary.tsv --overwrite

# QC
casetrack censor   --project-dir . --level LEVEL --id ID --kind qc_warn --reason "…"
casetrack uncensor --project-dir . --level LEVEL --id ID --reason "…"
casetrack qc-history --project-dir . --id ID

# querying & health
casetrack status   --project-dir .
casetrack query    --project-dir . --sql "SELECT … FROM _active WHERE …"
casetrack validate --project-dir .
casetrack doctor   --project-dir .
